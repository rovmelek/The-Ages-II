## What Went Well

### 1. Mechanical Refactors at Scale
Stories 14.1, 14.2, 14.4a, 14.4b, and 14.5 were large-blast-radius mechanical refactors (36 files in 14.4a alone). All completed with zero assertion value changes. Pattern: mechanical change → `make test` → verify.

### 2. Two-Phase Migration Pattern (14.4a)
PlayerSession dataclass migration used a compat bridge (`__getitem__`/`get`) in Phase 1, verified all tests passed, then migrated call sites in Phase 2. Zero-risk approach for a type-safety change touching 36 files.

### 3. Additive-Only Message Enrichment (14.3a, 14.3b)
Server messages enriched without removing or renaming existing fields. Web client used nullish coalescing fallbacks. Clean client/server decoupling.

### 4. Concurrency & DB Maturity (14.6, 14.7)
Async locks for TOCTOU races, Alembic migrations, connection pooling, timezone-aware datetimes, and transaction consolidation (3→1 per participant). Both transparent to existing code.

### 5. Adversarial Story Review Process
Multi-pass adversarial reviews caught real issues: wrong line numbers, incorrect table counts, missing PostgreSQL handling, false claims about datetime occurrences.

## Challenges

### 1. Test Debt Accumulated Across Stories
10 tests broke across 14.2, 14.3b, and 14.4b — carried forward as "pre-existing failures." Two collection errors from deleted `server.items.loot`, 8 assertion failures from changed types/fields. Not fixed in-story.

### 2. SQLite/PostgreSQL Impedance Mismatch
`DateTime(timezone=True)` behaves differently on SQLite vs PostgreSQL. Required `_ensure_aware()` helper — will recur until PostgreSQL is primary.

### 3. Dual Source of Truth for NPC Templates
`_NPC_TEMPLATES` module-level dict still exists alongside `game.npc_templates`. Not addressed.

## Key Patterns Established

- Pure mechanical refactor verified by `make test`
- Two-phase migration with compat bridge for type changes
- Additive-only message enrichment with client fallbacks
- Short lock critical sections (check-and-set only)
- `asyncio.gather` test pattern for concurrency verification
- Transaction consolidation with per-participant isolation
- Multi-pass adversarial story review before implementation
