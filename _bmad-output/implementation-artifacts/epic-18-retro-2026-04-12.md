## Epic Summary

Epic 18 transformed energy from a transient combat-only resource into a persistent player stat (like HP). The work covered four interconnected features: persistent energy derived from INT+WIS, card type classification (physical=free, magical=costs energy), XP bar progress display fix, and level-up point stacking. Critical discoveries included: flee-path stat sync bug (ISS-033), respawn DoT persistence bug (ISS-034), card_repo upsert gap, missing protocol entries, and the `_resolve_stats` migration bug for existing players.

2. **The quick-spec → quick-dev pipeline worked smoothly.** Despite bypassing the full epic/story/sprint-planning workflow, the tech spec was comprehensive enough to serve as a complete implementation guide. 33 tasks with 31 acceptance criteria left no ambiguity.

3. **15 ADRs provided clear rationale for every design decision.** Card classification rubric, energy formulas, regen architecture, existing player migration, combat sync, and more. Each ADR included rejected alternatives with reasoning.

4. **Existing codebase patterns absorbed the feature naturally.** `_STATS_WHITELIST`, effect registry plug-in, `settings.*` centralization, service layer architecture — all extended cleanly. Very little new abstraction needed.

## Challenges and Learnings

1. **Quick-dev bypassed sprint tracking.** Epic 18 has no entries in `sprint-status.yaml`, no individual story files, and no entry in `epics.md`. The closing process required backfilling. For future quick-dev epics, add at minimum an epic-level entry to sprint-status.yaml before starting.

2. **Card classification required multiple iterations.** `venom_fang` was reclassified 3 times (magical → physical → magical) before the rubric was formalized. The final rubric (physical = ALL effects are physical-damage/shield/DoT-with-physical-damage; magical = ANY non-physical effect) is clean, but arriving there required adversarial pressure.

3. **Adversarial reviews found diminishing returns after round 5.** Rounds 1-4 found critical issues (63 total). Rounds 5-7 found only 1 issue (ISS number collision). The sweet spot for this project is ~4-5 rounds.

4. **The tech spec grew very large (~530 lines).** After 7 review rounds, the spec accumulated decisions, notes, cross-references, and edge case documentation. While comprehensive, a fresh dev agent might find it overwhelming. Consider a "summary view" for implementation vs. the full decision record.

5. **`combat/service.py` has 4 separate stat-sync functions** (`sync_combat_stats`, `clean_player_combat_stats`, `cleanup_participant`, `handle_flee_outcome`). Each needed the same energy sync change. This is a code smell — consider extracting a shared `_sync_stats_from_combat()` helper in a future cleanup.

## Key Insights

1. **Quick-spec is viable for well-understood features.** When the codebase is mature and patterns are established, skipping the full epic/story workflow saves overhead. But sprint tracking should not be skipped — it breaks the closing process.

2. **Adversarial review is the highest-value quality gate.** 63 issues found before implementation. Many would have been runtime bugs or data loss (flee stat sync, existing player migration, card_repo upsert).

3. **The 8 project guidelines make excellent review criteria.** Structured adversarial review against "no game logic in client," "centralized config," "PostgreSQL ready," etc. caught issues that general review missed.

4. **Energy system designed for NPC extension (Epic 19).** The `card_type` field, energy formulas, and regen architecture are all extensible to NPCs when they gain card-based combat.

5. **Balance concerns are documented but deferred.** Physical builds have no resource decisions (all cards free). INT is double-loaded for casters. Energy potion trivializes energy at low levels. All flagged for playtesting — the settings-based formulas allow rebalancing without code changes.
