## What Went Well

### 1. "Extend, Don't Rewrite" (ADR-5) Validated

Story 12.6 audited CombatInstance and found it was already designed for N players — list-based participants, round-robin turns, dead-player skipping, per-player rewards. The entire story's production changes were one targeted bug fix, one convenience method (`start_combat()`), and one handler refactor. This validated the architectural decision from Epic 4-6 planning.

### 2. Consistent Handler and Manager Patterns

Every story followed identical structural patterns: manager class owned by `Game`, handler with `async def handle_X(ws, data, *, game)` signature, lambda closure registration, `TYPE_CHECKING` imports. Trade (12.1-12.2) established patterns that party (12.3-12.5) replicated mechanically. Direct payoff from the Epic 11 session factory refactor.

### 3. Effective Story Decomposition

- Trade: happy-path (12.1) + hardening (12.2)
- Party: infrastructure (12.3) + commands (12.4) + chat (12.5)
- Combat: multi-player engine (12.6) + party integration (12.7)
- Map: standalone (12.8)

Each could be reviewed and tested independently with clear dependency chains.

### 4. Disconnect Cleanup Order Managed Deliberately

`_cleanup_player` grew across 4 stories (12.1-12.4) with each adding a step in a specific position. The architecture doc specified the order (trades → combat → party → save → room → notify), and every story respected it. Zero ordering bugs.

### 5. Zero Regressions

Every story reported zero regressions across the full test suite. When pre-existing tests broke due to new hooks, they were fixed as part of the story implementation.

## Challenges and Lessons Learned

### 1. Test Fragility from Missing Mocks

Stories 12.2, 12.3, and 12.8 all had to fix tests because `_cleanup_player` gained new manager calls that existing tests didn't mock. The same test files were patched multiple times.

**Lesson:** When adding a new manager call to `_cleanup_player`, grep for all tests calling it upfront. Same lesson from Epic 11 — it recurred.

### 2. Dead Code from Spec Deviation

Story 12.4 placed invite tracking in module-level dicts rather than on the `Party` dataclass as specced. The `Party.pending_invites` field from 12.3 now sits unused.

**Lesson:** When deviating from spec, clean up unused scaffolding from previous stories.

### 3. Behavior Changes Buried in Feature Stories

Story 12.7 changed XP and loot behavior for ALL combat (dead players no longer get XP, loot is per-player independent rolls), but these were embedded in a party combat story.

**Lesson:** When a feature story changes existing behavior for all players, call it out prominently.

3. Incremental story decomposition with clear dependency chains enables effective testing and review.
4. The ISS bug-tracking process (ISS-014 through ISS-016) caught and resolved issues that would have compounded.
