## What Went Well

### 1. Clean Architectural Refactor
35 files modified with zero gameplay behavior changes. Every test passed after migration. The mechanical nature of the change (remove commit, rename call site) made it predictable despite the large blast radius.

### 2. The `flush()` Insight
Correctly identified during spec creation that `player_repo.create()` and `room_repo.upsert_room()` needed `session.flush()` instead of commit removal, preserving auto-increment ID generation without ending the transaction.

### 3. Bypass Pattern Elimination
All 4 repo bypass patterns replaced with clean repo calls inside `transaction()` blocks:
- Trade swap: raw `sa_update` → `player_repo.update_inventory()` × 2
- Combat loot: direct model mutation → `player_repo.update_inventory()`
- Chest: direct model mutation → `player_repo.update_inventory()`
- Scheduler: removed 2 explicit commits, let transaction CM handle it

### 4. ISS-017 Discovery
`Scheduler.stop()` was synchronous, causing flaky integration test failures from connection leaks during portal teardown. Discovered during implementation, fixed by making `stop()` async with proper task cancellation.

## Challenges

### 1. Test Mock Migration Scale
18 test files with 159 `session_factory` references required careful batching. No shortcuts — each file needed individual attention to determine mock vs real DB fixture.

### 2. Spec Gap: `room_repo.save_state()`
The adversarial spec review missed that `save_state()` uses `session.merge()` which returns a new object needing `flush()` for ID population. Discovered and fixed during implementation.

## Key Takeaway

The `Game.transaction()` pattern is now the single point of commit/rollback for the entire codebase. The server is PostgreSQL-ready with proper transaction boundaries, atomic multi-write operations, and automatic rollback on exceptions. The repo pattern rule changed from "repos commit internally" to "repos never commit — the transaction context manager handles it."
