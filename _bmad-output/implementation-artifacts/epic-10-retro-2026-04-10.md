## What Went Well

### 1. Clean Separation of Concerns
The epic naturally divided into server, client, and data stories. Client-only stories (10.3, 10.6) were fast and risk-free. Server stories (10.1, 10.4, 10.7) followed established patterns. Data-only stories (10.9) required no code changes at all.

### 2. Shared Module Extraction (10.7)
Moving `LOOT_TABLES` and `generate_loot()` from `chest.py` to a shared `server/items/loot.py` was a clean refactor that benefited both chest and combat systems. The dual-write pattern (DB + runtime inventory) was established correctly.

### 3. `_cleanup_player()` DRY Pattern (10.1)
Extracting the cleanup sequence into a shared helper prevented code duplication between logout, disconnect, and duplicate-login-kick paths. This reduced the surface area for bugs in player cleanup.

### 4. Extensible Slash Command Architecture (10.3)
The `COMMANDS` registry pattern made adding new commands trivial. Stories 10.6 added 4 more commands by simply inserting entries into the registry object — no structural changes needed.

### 5. Zero New Test Failures
Every story maintained the test baseline. The 2 pre-existing `TestChestInteraction` failures and known hanging tests were consistently documented and excluded, never confused with new regressions.

### 6. Comprehensive Adversarial Story Reviews
Stories went through multi-pass adversarial reviews before implementation, catching line number inaccuracies, coordinate conflicts, and missing context before any code was written.

## Challenges & Lessons Learned

### 1. Pre-existing Test Debt (`TestChestInteraction`)
The 2 `TestChestInteraction` DB session failures have persisted since Epic 7. They're consistently documented but never fixed. While they don't block development, they add noise to every test run and erode confidence in the test suite.

**Lesson:** Persistent test failures should be fixed or formally deferred with a tracking issue, not carried indefinitely.

### 2. Python Environment Complexity
Multiple stories noted that `pytest` must be run via `.venv/bin/python -m pytest` because the system Python (fbcode 3.12) lacks project dependencies. This caused friction in every story's test phase.

**Lesson:** Document the exact test command in CLAUDE.md (already done) and consider a Makefile or script alias.

### 3. Mock Pattern Discovery Curve (10.1, 10.4)
Stories 10.1 and 10.4 had to rediscover mock patterns (`new_callable=AsyncMock`, `__aenter__`/`__aexit__` setup, `Game.__init__` doesn't call `_register_handlers()`). These patterns were documented in story dev notes and propagated to later stories.

**Lesson:** The "Previous Story Learnings" section in story files was valuable for knowledge transfer between stories. Continue this pattern.

### 4. Empty Dict Falsy Gotcha (10.4)
`stats or default` silently skipped empty dicts because `{}` is falsy in Python. Fixed with explicit `is not None` check. This is a common Python footgun.

**Lesson:** Always use `is not None` for dict/list presence checks, never truthiness.

### 5. Large Room JSON Files (10.9)
The 100x100 room JSON files are single-line minified files (~30K tokens), making them unreadable by file-reading tools. Required Python scripts to extract spawn point data.

**Lesson:** Consider pretty-printing room JSON files for maintainability, or add a helper script for extracting room metadata.
